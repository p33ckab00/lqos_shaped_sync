LQoSync

Project Description

LQoSync is a focused LibreQoS companion tool. It is not a billing system, CRM, ISP Manager, or subscriber database. Its job is to read live MikroTik data, generate LibreQoS-compatible ShapedDevices.csv and network.json, then call LibreQoS.py --updateonly only when generated files changed or when a previous apply failed and must be retried.

Core rule: MikroTik and /opt/libreqos/src/config.json are the source of truth. LQoSync never writes to MikroTik and never stores subscribers in a database.
MikroTik read-onlyNo DBAtomic file writesBare-metal priorityLibreQoS-safe lifecycle

Process Workflow

1
Read live configuration and current LibreQoS files
Loads /opt/libreqos/src/config.json, ShapedDevices.csv, and network.json. Existing Circuit ID and Device ID are preserved when the same Circuit Name appears again.
2
Read MikroTik through read-only RouterOS API
Collectors read PPPoE secrets/active/profile, DHCP leases, and optionally Hotspot active sessions. LQoSync does not add, set, disable, delete, reboot, or modify anything on MikroTik.
3
Build proposed ShapedDevices and network layout
The engine resolves client names, MAC/IP, min/max speeds, speed source, parent nodes, network mode, node aggregation factors, and inactive cleanup rules.
4
Preflight and diff
Checks duplicate IPs, invalid bandwidth, missing parent nodes, JSON/CSV render validity, and computes added/updated/removed clients for preview, audit, and dashboard timeline.
5
Dry-run or apply
Dry-run only shows proposed output. Non-dry-run writes files atomically when changed, then triggers LibreQoS apply based on the apply policy.
6
Record result
Runtime state, audit events, apply stdout/stderr, elapsed timings, client-change info bits, and scheduler status are saved for the UI.

LibreQoS Apply Policy

The production default is automatic apply when files change. If dry-run is used, LQoSync must never write or apply. If LibreQoS apply fails after files were written, LQoSync marks a pending apply and retries on the next non-dry-run cycle.

"libreqos": { "cmd": "/opt/libreqos/src/LibreQoS.py", "args": ["--updateonly"], "working_dir": "/opt/libreqos/src", "run_mode": "direct", "sudo": true, "run_only_when_files_changed": true, "retry_if_last_apply_failed": true, "timeout_seconds": 300 }
Why working_dir matters: LibreQoS loads files like ShapedDevices.csv and ShapedDevices.lastLoaded.csv by relative name. Running from /opt/libreqos/src prevents FileNotFoundError.
Bare-metal priority: Bare-metal installs must use run_mode=direct. Docker-only host_nsenter must not survive on systemd installs.

Selective MikroTik Collection and Speed Resolver

LQoSync v2.38 adds a future-proof collector model for hundreds to thousands of clients. The goal is not to make many per-client API calls. The goal is to bulk-read each RouterOS source, request only the properties needed for LibreQoS output, index locally in memory, and resolve speed from a universal priority chain.

PPPoE

Reads /ppp/active, /ppp/secret, and /ppp/profile with selected fields.

Speed priority: 1 secret comment 2 active comment 3 profile comment 4 profile name 5 profile rate-limit 6 default_pppoe_rate
DHCP

Reads /ip/dhcp-server/lease and DHCP server name/comment metadata. Dynamic lease comments are not used as the first priority because they are unreliable unless the lease is static.

Speed priority: 1 DHCP server comment / speed_comment 2 DHCP server name 3 DHCP server config speed 4 global default DHCP speed
Hotspot

Reads /ip/hotspot/active. When enhanced metadata is enabled, also reads /ip/hotspot/user and /ip/hotspot/user/profile.

Speed priority: 1 user comment 2 profile comment 3 profile name 4 profile rate-limit 5 hotspot config speed 6 global default

Collector config

"collector": { "selective_fields": true, "pppoe": { "read_active": true, "read_secrets": true, "read_profiles": true }, "dhcp": { "lease_mode": "permissive", "read_server_metadata": true }, "hotspot": { "enhanced_metadata": true } }
Selective fields: LQoSync asks RouterOS for only fields needed by ShapedDevices.csv and network.json. If a RouterOS/API combination rejects selective property reads, LQoSync falls back to full reads to preserve compatibility.
Metadata cache: Source hashes and metadata-cache status are saved in /opt/lqosync/state/collector_cache.json. It is a lightweight state file, not a database. Dashboard cards show cache hits, misses, and source changes.

Dashboard monitoring added

The Dashboard now exposes Collector Health, Source Counts, Speed Source Breakdown, Cache Efficiency, and richer Last Sync Timeline entries. This helps detect whether PPPoE, DHCP, Hotspot, speed resolution, cache, or LibreQoS apply is the current bottleneck.

Fresh Installation

Fresh install means LibreQoS exists but some managed files may be missing. LQoSync should dynamically create missing files from templates while protecting existing LibreQoS files.

GitHub source install

sudo apt update sudo apt install -y git cd /opt sudo git clone https://github.com/p33ckab00/lqos_shaped_sync.git lqosync cd /opt/lqosync sudo bash install.sh

One-command bootstrap

curl -fsSL https://raw.githubusercontent.com/p33ckab00/lqos_shaped_sync/main/install-from-github.sh -o /tmp/install-lqosync.sh sudo bash /tmp/install-lqosync.sh

Smart init behavior

Fresh LibreQoS install: missing config.json / ShapedDevices.csv / network.json -> create from templates Existing/live LibreQoS install: files exist -> preserve by default or ask operator Non-interactive install: files exist -> preserve existing by default
Before enabling scheduler: stop old writers such as updatecsv.service so two systems do not write the same LibreQoS files.

Existing Install Adoption

If /opt/lqosync already exists, the GitHub installer should not delete it blindly. It should detect how the system was installed and ask how to proceed.

GitHub Updates and Preservation Policies

Git updates the application source. Migration updates only missing safe config keys. Operator-owned config and runtime files are preserved by default.

Default production update

cd /opt/lqosync sudo bash upgrade.sh

Default policy is preserve_and_migrate: pull code, preserve live files, add missing safe defaults, reapply permissions, restart, and health check.

Explicit policy examples

sudo UPDATE_POLICY=pull_only bash upgrade.sh sudo UPDATE_POLICY=code_only bash upgrade.sh sudo UPDATE_POLICY=preserve_and_migrate bash upgrade.sh sudo UPDATE_POLICY=refresh_with_backup bash upgrade.sh sudo UPDATE_POLICY=factory_reset CONFIRM_FACTORY_RESET=yes bash upgrade.sh

Policy meanings

PolicyPurpose
pull_onlyOnly fetch/pull code. No migration, dependency update, or restart. Use when reviewing changes first.
code_onlyUpdate app source but avoid touching config/users/generated files. Good for UI-only fixes.
preserve_and_migrateProduction default. Preserve local data and add missing safe defaults.
refresh_with_backupBackup then refresh service files, dependencies, permissions, and migrations. Good for major updates or repair.
factory_resetDanger mode. Requires explicit confirmation. Intended for lab rebuilds or broken config recovery.

Uninstallation and Permission Restore

Uninstall should remove LQoSync while preserving LibreQoS integrity. It should stop the service, remove LQoSync service/sudoers, optionally remove /opt/lqosync, and restore permissions under /opt/libreqos/src.

Standard uninstall

cd /opt/lqosync sudo bash uninstall.sh

Restores LQoSync-managed LibreQoS files to root ownership and removes stale ACLs.

Remove runtime too

cd /opt/lqosync sudo REMOVE_RUNTIME=true bash uninstall.sh

Backs up and removes /opt/lqosync after service cleanup.

Full permission restore
cd /opt/lqosync sudo RESTORE_MODE=full bash uninstall.sh # or directly sudo bash /opt/lqosync/scripts/restore_libreqos_permissions.sh --full
Do not delete by default: config.json, ShapedDevices.csv, network.json, ShapedDevices.lastLoaded.csv, lastGoodConfig.csv, lastGoodConfig.json, and planner_state.json are LibreQoS operational files.

MikroTik Setup Requirement

Important Notice: For fresh installations, create a dedicated read-only RouterOS API user before enabling live sync. LQoSync reads MikroTik only; it should not use an admin/write-capable account.

RouterOS terminal commands

/user group add name=API_READ policy="read,sensitive,api,!policy,!local,!telnet,!ssh,!ftp,!reboot,!write,!test,!winbox,!password,!web,!sniff,!romon" /user add name="libreqosyncAPI" group=API_READ password="<Strong Password>" address="<LibreQoS IP Address>" disabled=no

What this protects

  • Allows the required RouterOS API reads for PPPoE, DHCP, Hotspot, and profile discovery.
  • Blocks write, reboot, policy, web, Winbox, sniffing, and RoMON access.
  • Restricts login source using address=<LibreQoS IP Address>.
  • Use this user in /opt/libreqos/src/config.json.

Troubleshooting Guide with Expected Results

This section explains what each troubleshooting step does, why it is safe, and what to expect.

LibreQoS failed: FileNotFoundError: ShapedDevices.csv

Meaning: LibreQoS.py was executed from the wrong working directory and searched for ShapedDevices.csv relative to that directory.

grep -A12 '"libreqos"' /opt/libreqos/src/config.json # Expected: working_dir=/opt/libreqos/src and run_mode=direct
latest_json=$(sudo find /opt/lqosync/logs/libreqos_apply -name "*.json" -printf '%T@ %p\n' | sort -n | tail -1 | cut -d' ' -f2-) sudo cat "$latest_json" # Expected: "working_dir": "/opt/libreqos/src"

Fix expectation: after updating config or reinstalling, the journal should show PWD=/opt/libreqos/src for LibreQoS apply commands.

nsenter: cannot open /proc/1/ns/ipc: Permission denied

Meaning: Docker host namespace mode leaked into a bare-metal install. Bare-metal must use direct execution.

grep -E 'LQOSYNC_RUN_MODE|HOST_CONTROL_MODE|LQOSYNC_INSTALL_MODE|LQOSYNC_FORCE_DIRECT' /opt/lqosync/.env # Expected: # LQOSYNC_INSTALL_MODE=baremetal # LQOSYNC_RUN_MODE=direct # HOST_CONTROL_MODE=direct # LQOSYNC_FORCE_DIRECT=true
sudo systemctl restart lqos_shaped_sync

Expected result: future apply metadata shows mode=direct, not host_nsenter.

Permission denied: /opt/libreqos/src/config.json.tmp

Meaning: LQoSync can read the config but cannot create atomic temporary files in /opt/libreqos/src.

sudo apt update sudo apt install -y acl sudo setfacl -m u:lqosync:rwx /opt/libreqos/src sudo setfacl -m u:lqosync:rw /opt/libreqos/src/config.json sudo setfacl -m u:lqosync:rw /opt/libreqos/src/ShapedDevices.csv sudo setfacl -m u:lqosync:rw /opt/libreqos/src/network.json sudo setfacl -d -m u:lqosync:rwX /opt/libreqos/src
sudo -u lqosync touch /opt/libreqos/src/config.json.tmp sudo -u lqosync rm -f /opt/libreqos/src/config.json.tmp # Expected: no error

What happens: ACL grants the lqosync service user permission to create temporary files and atomically rename them without changing the entire LibreQoS directory ownership.

files_changed=False but LibreQoS was not applied after a previous failure

Meaning: files were already written, so the next diff is clean. The missing behavior is retrying a pending failed apply.

grep -A12 '"libreqos"' /opt/libreqos/src/config.json # Expected: "retry_if_last_apply_failed": true

Expected result: after the fix, non-dry-run cycles retry LibreQoS apply even if file content did not change, until apply succeeds.

LibreQoS status shows false in UI but apply logs are OK

Meaning: apply engine may be healthy while the service monitor/UI parsing is wrong or one optional service is missing.

systemctl is-active lqosd systemctl is-active lqos_scheduler systemctl status lqosd lqos_scheduler --no-pager

Expected: lqosd and lqos_scheduler should be active. lqos_node_manager is legacy/optional and should not make current LibreQoS look failed if missing.

Blank /var/log/lqos_shaped_sync.log

Meaning: current config may write logs under /opt/lqosync/logs and systemd journal instead.

tail -n 100 /opt/lqosync/logs/lqos_shaped_sync.log journalctl -u lqos_shaped_sync -n 100 --no-pager

Expected: application sync logs are in /opt/lqosync/logs/lqos_shaped_sync.log; service lifecycle logs are in journalctl.

git pull does not work: not a git repository

Meaning: the installation came from ZIP/manual copy. Use the adoption installer to make it Git-managed without losing live data.

cd /opt curl -fsSL https://raw.githubusercontent.com/p33ckab00/lqos_shaped_sync/main/install-from-github.sh -o /tmp/install-lqosync.sh sudo EXISTING_INSTALL_ACTION=adopt bash /tmp/install-lqosync.sh

Expected: the installer backs up current /opt/lqosync, preserves users/logs/state/config, and installs repo code into a Git-managed /opt/lqosync.

Port 9202 is already in use
sudo ss -tulpn | grep :9202 sudo systemctl status lqos_shaped_sync

Meaning: another process is listening on the UI port. Stop the duplicate process or change the port in the systemd/env config.

ShapedDevices/network validation fails
cd /opt/libreqos/src head -1 ShapedDevices.csv awk -F',' 'NR>1 && $5=="" {print NR ":" $0}' ShapedDevices.csv python3 -m json.tool network.json

Expected: header is valid, parent nodes are present unless using flat no-parent mode, and network.json parses cleanly.

v2.45 Smart Policy Center

Purpose: LQoSync now evaluates policy before writing ShapedDevices.csv, writing network.json, or running LibreQoS.py --updateonly. This prevents blind cleanup and makes cleanup/apply decisions explainable in Dashboard and Dry Run.

Architecture: Config defines operator intent. Policies define how safely that intent is applied. The policy engine decides before write/apply. Dashboard and Dry Run explain every decision.

Policy decision flow

1. Load config and existing files 2. Collect MikroTik PPP/DHCP/Hotspot data 3. Build proposed output in memory 4. Preview stale rows and cleanup candidates 5. Evaluate cleanup/source/apply policies 6. Run preflight validation 7. Block / warn / require confirmation / allow 8. Only then write files and optionally apply LibreQoS

Cleanup actions

Supported actions are preserve_rows, warn_only, cleanup_immediate, cleanup_next_run, require_confirm_immediate, require_confirm_next_run, block_cleanup, and block_apply.

Important protections

  • Collector failure preserves rows by default.
  • Enabled source returning zero rows can block cleanup.
  • Source-disabled cleanup can require operator confirmation.
  • Duplicate IP, missing parent, invalid speed, and collector failure can block write/apply.
  • Queued cleanup and pending confirmations are stored in /opt/lqosync/state/policy_state.json.

Operator pages

Use Policy Center to review pending confirmations, source cleanup policies, apply guards, collector guards, runtime policy state, and the last policy decision. Use Dry Run to see the same verdict without writing files or applying LibreQoS.
